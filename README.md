# Frontend Project Specification for AI Revenue Engine

This file is a project specification for Claude Code to generate a complete Angular frontend app.
The frontend must use normal CSS, Bootstrap, and FontAwesome icons, and it must not use Angular standalone components.

## Project summary

Build a full Angular UI that connects to the existing FastAPI backend at `http://localhost:8000`.
The frontend must include:
- Authentication
- Dashboard metrics
- Lead management
- Analytics reporting
- Workflow management
- Admin / feature flags / prompt management
- Realtime stream support via SSE and WebSocket

## Stack details

- Angular application generated by Angular CLI
- Routing enabled
- `BrowserModule`, `HttpClientModule`, `ReactiveFormsModule` used
- Bootstrap for layout and style
- FontAwesome for icons
- Standard component + module architecture (no standalone components)
- CSS files for custom styling

## Project structure

The generated project should follow this structure under `frontend/src/app`:

- `app.module.ts`
- `app-routing.module.ts`
- `core/`
  - `auth.service.ts`
  - `api.service.ts`
  - `token.interceptor.ts`
  - `auth.guard.ts`
- `pages/`
  - `login/`
  - `register/`
  - `dashboard/`
  - `leads/`
  - `analytics/`
  - `workflows/`
  - `admin/`
- `shared/`
  - `navbar/`
  - `card/`
  - `table/`
- `styles.css`

## UI requirements

The frontend should use Bootstrap components for:
- page layout and responsive grid
- navigation bar
- cards and panels
- forms and buttons
- tables and badges
- alerts and status indicators

Include FontAwesome icons on:
- navigation items
- cards and headings
- buttons and actions
- status badges and notifications

Use CSS files for component-specific styling and a global `styles.css`.

## Routing requirements

The app must support these routes:
- `/login`
- `/register`
- `/dashboard`
- `/leads`
- `/leads/:id`
- `/analytics`
- `/workflows`
- `/admin`
- default redirect to `/dashboard`
- wildcard redirect to `/dashboard`

Protected routes must require authentication and redirect to `/login` if unauthenticated.

## Authentication flow

The frontend must implement:
- login page
- register page
- token storage in browser storage
- HTTP interceptor to attach `Authorization: Bearer <token>` header to requests
- route guard for protected pages
- logout behavior

## Page requirements

### Login page

- email and password fields
- submit button
- error display for invalid credentials
- Bootstrap centered card layout
- FontAwesome login icon in header or button

### Register page

- email, password, tenant name fields
- submit button
- Bootstrap form styling
- FontAwesome registration icon

### Dashboard page

- KPI cards for total leads, conversion rate, funnel progress, and recent events
- funnel summary or chart area
- call-to-action cards for quick lead ingestion or workflow creation
- FontAwesome icon for each card

### Leads page

- table of leads with columns: name, score, stage, source, assigned agent
- filter controls for stage and score
- clickable rows to open lead details
- badges for score and stage values
- action icons for view, refresh, and trigger events

### Lead detail page

- lead profile information
- score, intent, stage, source, message text
- follow-up status summary
- stage update controls
- behavioral trigger controls
- section headings with FontAwesome icons

### Analytics page

- overview section with key metrics
- funnel results section
- agent performance summary
- advanced analytics summary card or link
- Bootstrap cards and icons for each section

### Workflows page

- list of workflows with name, trigger, steps, and status
- buttons for add, edit, delete, and test workflow
- FontAwesome icons for workflow actions
- form for workflow details and step configuration

### Admin page

- feature flags panel
- dynamic config panel
- prompt registry panel
- usage and quota panel
- next best action and orchestrator admin controls
- buttons with icons for save, rollback, top-up, and refresh

### Realtime panel

- SSE or WebSocket event feed
- connection status display
- live event list with timestamps
- event type icons

## Realtime integration

The frontend should support both:
- SSE: `GET /stream/dashboard?token=<JWT>`
- WebSocket: `WS /ws/leads?token=<JWT>`

The realtime consumer should display live lead events and metrics.

## Backend API list

The frontend must call these backend APIs directly or via services.
All protected endpoints require `Authorization: Bearer <token>`.

### Authentication

- `POST /auth/login`
  - request body:
    ```json
    {
      "email": "user@example.com",
      "password": "secret"
    }
    ```
  - response body:
    ```json
    {
      "access_token": "eyJ...",
      "token_type": "bearer"
    }
    ```
  - purpose: log in user and receive a JWT token

- `POST /auth/register`
  - request body:
    ```json
    {
      "email": "user@example.com",
      "password": "secret",
      "tenant_name": "My Company"
    }
    ```
  - response body:
    ```json
    {
      "access_token": "eyJ...",
      "token_type": "bearer"
    }
    ```
  - purpose: create a new user and tenant

### Leads

- `POST /leads/ingest`
  - request body:
    ```json
    {
      "message": "I need CRM for 20 users",
      "source": "whatsapp",
      "phone": "+919876543210",
      "email": "karthik@startup.io"
    }
    ```
  - response body example:
    ```json
    {
      "id": "uuid",
      "name": "Karthik",
      "score": "high",
      "intent": "buying",
      "stage": "qualified",
      "assigned_agent": "sales_agent",
      "opening_message": "Karthik, a CRM that scales...",
      "followup_tasks_scheduled": 2
    }
    ```
  - purpose: create a new lead from an input message

- `GET /leads`
  - query parameters: `stage`, `score`, `limit`, `offset`
  - response body example:
    ```json
    [
      {
        "id": "uuid",
        "name": "Karthik",
        "score": "high",
        "stage": "qualified",
        "source": "whatsapp",
        "assigned_agent": "sales_agent",
        "created_at": "2025-01-01T12:00:00Z"
      }
    ]
    ```
  - purpose: list leads with optional filters

- `GET /leads/{id}`
  - response body example:
    ```json
    {
      "id": "uuid",
      "name": "Karthik",
      "message": "I need CRM for 20 users",
      "score": "high",
      "intent": "buying",
      "stage": "qualified",
      "source": "whatsapp",
      "phone": "+919876543210",
      "email": "karthik@startup.io",
      "assigned_agent": "sales_agent",
      "followup_tasks_scheduled": 2,
      "created_at": "2025-01-01T12:00:00Z"
    }
    ```
  - purpose: retrieve a single lead detail

- `PATCH /leads/{id}/stage`
  - request body:
    ```json
    {
      "stage": "converted"
    }
    ```
  - response body example:
    ```json
    {
      "id": "uuid",
      "stage": "converted"
    }
    ```
  - purpose: update lead pipeline stage

- `POST /leads/trigger`
  - request body:
    ```json
    {
      "lead_id": "uuid",
      "event": "link_click",
      "metadata": {
        "url": "https://example.com"
      }
    }
    ```
  - response body example:
    ```json
    {
      "scheduled": true,
      "task": "task-id"
    }
    ```
  - purpose: send a behavioral trigger event for a lead

### Realtime streams

- `GET /stream/dashboard?token=<JWT>`
  - response: server-sent event stream with event types such as `lead_created`, `metrics`, and `lead_scored`

- `GET /stream/leads?token=<JWT>`
  - response: server-sent event stream with lead lifecycle events

- `GET /stream/metrics?token=<JWT>`
  - response: server-sent event stream with metrics snapshots

- `WS /ws/leads?token=<JWT>`
  - response: WebSocket events encoded as JSON messages for live lead feed

## Technical implementation notes

- Use `http://localhost:8000` as the API base URL.
- All protected requests must include `Authorization: Bearer <token>`.
- Use an `AuthGuard` to protect routes.
- Use an HTTP interceptor to inject the JWT token for all outgoing requests.
- Use Bootstrap layout utilities (`container`, `row`, `col-`, `d-flex`, `me-`, `mb-`, etc.).
- Use FontAwesome icons such as `fas fa-user`, `fas fa-chart-line`, `fas fa-cogs`, `fas fa-bell`, `fas fa-stream`, and `fas fa-rocket`.
- Keep the app responsive.
